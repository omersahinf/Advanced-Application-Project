Executive Summary

The Flower E-Commerce Analytics Platform is a production-grade, three-tier web application that combines a relational e-commerce backend with a natural-language analytics assistant. The system serves three role-segregated audiences — Admin, Corporate, and Individual users — across a Stripe-backed checkout pipeline, a corporate KPI dashboard, an admin governance console, and a chatbot that converts plain-English questions into role-scoped SQL queries against the live database.

The chatbot is implemented as a six-agent LangGraph state machine (Guardrails → SQL Generator → Executor → Error Handler → Analyst → Visualizer) and enforces role-based query scoping through deterministic SQL filters rather than relying on the LLM's compliance. The design directly addresses the twelve attack vectors documented in the CSE 214 Security Penetration Test Cases catalogue, including prompt-injection, cross-corporate access, SQL-injection through the natural-language interface, JWT tampering, and stored XSS via review propagation.

The codebase consists of a Spring Boot 3 backend (Java 17), an Angular 21 frontend with the Flower design system, and a Python 3.11 chatbot service. Deployment is provided through Dockerfiles for each service plus a one-command docker compose up stack, with reference Kubernetes manifests included. Continuous integration runs on every push via GitHub Actions, executing 62 backend tests, 38 frontend tests, and the full chatbot pytest suite (35+ regression tests for role isolation and security).

1.3 Why Two Server Processes

Splitting the chatbot into a dedicated FastAPI process — rather than embedding LangGraph inside Spring Boot — yields three concrete benefits:

  • Language fit: LangGraph, Pandas, and the Python LLM ecosystem are first-class in Python; embedding them in the JVM would require duplication or fragile bridges.
  • Blast-radius isolation: any chatbot crash, runaway prompt, or LLM outage cannot take the e-commerce backend down; the user can still browse and check out.
  • Privilege separation: the chatbot connects to a separate, SELECT-only database role and authenticates to the backend with a shared CHATBOT_API_KEY — the chatbot literally cannot perform writes.

1.4 Role-Based Access Control

Every authenticated request carries an HS512-signed JWT containing the server-issued role. Authorization is layered: (i) a Spring Security filter rejects unsigned or tampered tokens; (ii) route-level hasAuthority(...) rules on SecurityConfig; (iii) service-level ownership checks (e.g. order.user_id == sessionUserId); and (iv) at the chatbot layer, deterministic SQL filters rewrite WHERE clauses to enforce the same scope — even if the LLM generated something broader.

2. Database Design

2.1 Entity-Relationship Model

The schema is normalized to 3NF with one deliberate denormalization (customer_profiles.total_spend), documented in DataSeeder.java. Every mutating entity carries created_at and updated_at audit timestamps populated through JPA @PrePersist / @PreUpdate lifecycle callbacks.

2.2 Index Strategy

TableIndexUse Case
orders(user_id, order_date)Individual dashboard / "my orders" listing
orders(store_id, order_date)Corporate sales-over-time KPIs
orders(status)Admin pipeline view; chatbot status filters
products(store_id), (category_id), (sku)Catalog browsing + analytics drill-down
reviews(product_id), (star_rating), (sentiment)Product page aggregation; sentiment dashboards

2.3 Data Integrity Constraints

  • CHECK (stock >= 0) on products
  • CHECK (star_rating BETWEEN 1 AND 5) on reviews
  • CHECK (grand_total >= 0) on orders
  • CHECK (cost_price IS NULL OR cost_price >= 0) on products
  • Cascade deletes on owned collections (Order → OrderItem, Order → Shipment) via CascadeType.ALL + orphanRemoval.

2.4 Audit & Soft State

All transactional entities (User, Store, Category, Product, Order, Shipment, Review, CustomerProfile) carry created_at and updated_at timestamps. The audit_logs table records administrator actions: every endpoint in AdminController that mutates user state, store status, category taxonomy, or platform settings is wrapped with an auditLogService.log(...) call.

This guarantees the demo dataset is identical on every fresh boot.

3.2 Transformations Applied

  • Field mapping: documented per-dataset in ETL_FIELD_MAPPING.md.
  • ID generation: surrogate BIGINT primary keys via Hibernate GenerationType.IDENTITY.
  • Date normalization: all timestamps stored as java.time.LocalDateTime in ISO-8601 format.
  • Currency: single-currency DECIMAL(12,2) with non-negative CHECK constraints; cost-price stored separately for margin analytics.
  • Sentiment derivation: star-rating mapped to POSITIVE / NEUTRAL / NEGATIVE at seed time.

4.2 Response Envelope & Errors

Successful responses return the resource DTO directly. Errors flow through GlobalExceptionHandler, which translates domain exceptions to safe HTTP responses without leaking stack traces or SQL fragments: 

ResourceNotFoundException        → 404
BadRequestException              → 400
UnauthorizedOperationException   → 403
AuthenticationException          → 401
MethodArgumentNotValidException  → 400 with field-level messages

4.3 Validation

All write endpoints accept dedicated DTOs annotated with Jakarta Bean Validation: @NotBlank, @Email, @Size, @Min, @Pattern. The custom InputValidator applies regex blocklists for prompt-injection / SQL-injection patterns before any natural-language input reaches the chatbot proxy.

4.4 Pagination & Filtering

List endpoints accept page, size, sort, search, and domain-specific filters (e.g. categoryId, status). Spring Data JPA's Pageable abstraction is used uniformly, returning Page<T> with totals and metadata.

5. AI Chatbot Architecture

5.1 Six-Agent LangGraph Pipeline

The chatbot is a deterministic state machine — not a free-form agent loop — orchestrated through LangGraph. Each node has a single responsibility, a well-defined input/output, and an explicit transition policy. The role context is injected once at request entry from the authenticated session and re-asserted in every subsequent SQL generation pass; the LLM never controls scoping.

5.2 Per-Agent Responsibilities

AgentRoleHighlights
GuardrailsScope & safetyDetects prompt injection, cross-user / third-person enumeration, greeting/out-of-scope intents.
SQL GeneratorNL → SQL~23 deterministic templates first; LLM fallback only for unmatched questions; role-scoped WHERE clauses are appended programmatically.
ExecutorSafe SQL executionSQLAlchemy text(); SELECT/WITH only; UNION, multi-statements, DDL, DML rejected; SET TRANSACTION READ ONLY.
Error HandlerRecoveryBounded retries with corrective hints fed back into SQL Generator.
AnalystResult narrationPandas-driven summary with optional LLM polish; deterministic when API key absent.
VisualizerPlotly chart codeAST validator blocks imports, dunders, eval, exec; rendered inside a sandboxed iframe.

5.3 Role-Scoped Query Synthesis

Role enforcement is applied at three layers, on every request:

  • Backend: the Spring Boot proxy attaches the authenticated userId, role, and (for corporate) storeId to the request body, server-to-server, with the shared CHATBOT_API_KEY header.
  • SQL Generator: regex injection rewrites WHERE clauses; for INDIVIDUAL adds user_id = ?, for CORPORATE forces store_id IN (?), ADMIN unrestricted but logged.
  • Executor: the database connection is opened on a SELECT-only role with sensitive columns blocked at the application layer (password_hash, api_key, internal_cost).

6.2 Defense-in-Depth Layers

  1. JWT signature verification (HS512, 64-char minimum secret enforced).
  2. Spring Security route-level role guards.
  3. Service-level ownership checks.
  4. Bean Validation on every DTO.
  5. Input regex blocklists for prompt-injection signatures.
  6. Chatbot guardrails agent (intent / scope / cross-user detection).
  7. Deterministic SQL templates with hard-coded role filters.
  8. Database connection on a SELECT-only role with READ ONLY transactions.
  9. Sensitive-column denylist applied at result-set level before return.

6.3 Secret Hygiene

  • JWT_SECRET required to be ≥ 64 characters; application fails on startup otherwise.
  • Stripe secret key only in backend; frontend receives publishable key + per-intent client secret.
  • Gemini API key never reaches the browser — all LLM traffic is server-to-server.
  • docker-compose.yml uses ${JWT_SECRET:?...} to refuse to start without an explicit secret.
  • Audit logs persist every admin action with actor, target, timestamp.

9. Challenges & Solutions

9.1 Trustworthy Multi-Agent Orchestration

Challenge: a free-form agent loop is hard to test, hard to debug, and unpredictable under adversarial prompts. Solution: the chatbot is implemented as a deterministic LangGraph state machine where each node is unit-testable in isolation. The bulk of demo prompts are answered by ~23 deterministic SQL templates; the LLM is a fallback for unmatched questions, never a controller of scope.

9.2 Role Scoping Without Trusting the LLM

Challenge: a corporate user could ask "show all stores' revenue" and the LLM might comply. Solution: WHERE clauses for ownership are appended programmatically by sql_generator.py after the LLM (or template) produces a query. The Guardrails agent additionally detects third-person enumeration ("how much has Bob spent") and refuses before reaching SQL.

9.3 JWT in Cookie vs Bearer Header

Challenge: storing access tokens in localStorage exposes them to XSS exfiltration; relying solely on cookies breaks Postman / curl flows. Solution: the backend issues HttpOnly SameSite=Strict cookies and accepts the same token in an Authorization: Bearer header. Tests were updated to read JWT from cookie metadata via MockMvc.

9.4 Plotly Without Eval

Challenge: rendering LLM-produced visualization code without enabling client-side script execution. Solution: the Visualizer agent emits Python code, runs it through an AST validator (no import, no dunders, no eval/exec), executes it in a restricted-builtins sandbox to produce static Plotly HTML, and the Angular frontend renders that HTML inside a sandboxed iframe. The frontend never calls eval or Function().

9.5 Chatbot CI Without Live Postgres

Challenge: a subset of chatbot regression tests originally required a live Postgres instance, which made CI brittle. Solution: those tests were rewritten to use an isolated SQLite fixture seeded with a minimal schema. The full pytest suite now runs in CI without external services and the role-scoped security regression tests run on every push.

9.6 Deterministic Demo Stability

Challenge: early in development, prompts like "how much have I spent this year?" produced LLM-generated SQL that occasionally truncated date filters or misnamed columns. Solution: these high-frequency prompts now have dedicated deterministic builders in sql_generator.py: time windows derive from DATE_TRUNC('year', CURRENT_DATE), the user scope is hard-coded, and cancelled orders are excluded by default.

10. Future Work

  • WebSocket notifications: the platform currently uses Server-Sent Events for streaming chatbot output. Migrating to a STOMP-over-WebSocket channel would unlock bidirectional features (typing indicators, multi-user dashboards).
  • Real CSV ingestion pipeline: while the seeder is sufficient for the demo, a Spring Batch / Pandas job that ingests the original Kaggle CSVs would tighten the ETL story.
  • Vector retrieval for chatbot: embedding product descriptions and review text into a vector index would let the SQL Generator handle truly novel questions.
  • OpenTelemetry traces: structured per-agent traces for observability of the chatbot pipeline.
  • End-to-end Playwright suite: visual regression for the three role surfaces.
